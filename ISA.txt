Notes
$# Means Hex Value (Followed By Number In Actual Assembly)
$ Means Register (Followed By A Valid Register Number In Assembly Or Working Register Letter)
# Means Binary Data
For Storing Registers Assume The Same
SBMP Stands For Sixteen Bit Micro Processor
If You Make Any Piece Of Software For This Please Release Both The Compiled Version, A Copy Of The Compile Command, The Source Code

Working Registers
A, B And C - 16 Bit Registers
X And Y - 8 Bit Registers

Iteration Registers
I (16 Bit Default)
Skip To Iteration In ISA For How To Use

- When 0 (Default) The Overflow Is Ignored
- ($08 Bit 3)

Iteration
CLI (No Parameters Clears I To 0)
ITR (Iterates I By 1 Checks If It Passes Limit And If It Does It Resets)
Iteration Limit Register ($07) (Sets The Maximum Value Of I Before It Becomes Zero)
Finished Iteration Register ($08 Bit 1) (When I Resets To 0 Set To 1) (After Read Reset To 0) (1 Bit Flag)

Branches
Set Position For Branching Added To List In Compiled ROM Format Of "BranchName:"
Branching reads from register $01, or the A register for the branch condition.
Note Branch Names Are Case Sensitive And Must Be Followed By The Colon
BIZ $ Or "BrancName:" (Branch If Zero)
BIO $ Or "BranchName:"(Branch If One)

---- Audio ----
Both Audio Commands Take In The Same Parameters Of 8 Bits For Frequency And 8 Bits FOr Length Of Time In cs (Centi Seconds)
ADN Audio Play With No Wait (Runs The Next Command Immediatly)
ADW Audio Play With Wait (Runs The Next Command After The Noise Has Been Played)
Audio Frequencies
First 32 Frequencies (0x00 - 0x20) Are Different Drums
Next 88 Frequencies (0x21 To 0x78) NES Style Chiptune From A0 To C8
Next 88 Frequencies (0x79 To 0xD0) Chip Piano (Triangle Wave) From A0 To C8
Remaining 48 Frequencies (0xD0 To 0xFF) Programmable Frequencies
---- End Of Audio ----

8 Highest

TRRRRRGGGGGBBBBB

---- Video Display ----
There Are 2 Modes Of Data Display In The PPU And It Is Set In Main $08 Bit 6
The Modes Are Bitmap And Tilemap

-- Tilemap Mode--
Note Tilemap Uses 1 Main Map For Foreground And Background Objects With A Secondary Map For Sprite Map 
-- Tile Maps --
Reg $01 - $16384 Map Of 256 Tiles For Blocks Type 16
Reg $16385 - $17409 Map Of 16 Tiles For Player Sprites
Screen Type 16
-- Layering --
The 3 Layers Are BG FG And SL (Background Foreground And Sprite Layer)
The Size Of Each Layer Depends On Main Reg $08 Bit 8,9 Setting
BG And FG Reference The Main Map With A Full Byte Of Data For Each Tile (The 256 Combinations)
SL Or Sprite Layer Is Only 1 Nibble Per Tile (16 Combinations) So W/16 For 8 Because Of Tile Map And 2 Because There Can Be 2 Tiles Referenced To Register Then H/8 Again Because 8*8 Tiles
The Final Size Of This Registers Should Be 2((W*H)/64)+(W*H)/128 For Total Amount Of Registers

-- Bitmap Mode --
It's Quite Simple But Not As Memory Efficient As Tilemap
Starts At $01
Using 16 Bit Registers Per Piece Of Data Going Left To Right Then Down However This Is Upscaled So 2*1 Pixels As That Would Otherwise Need Upto 31k Registers Which Is Alot And Breaks My 20k PPU Register Limit
Register Calculation Is W*H

-- Final Colour Output --
These Only Make 31 Which Isn't In Our 255,255,255 Standard Today So Instead We Multiply By The Brightness Setting Where The Max Is 8 Giving An Actual Colour Roof Being 248

------ Emulation ------
smbpe.exe can be used in two different modes, raw and file mode.

Raw mode is used for testing small snippets of assembly code. It takes the assembly code as input, compiles it, and runs it in the emulator. This mode is useful for quick tests and debugging.
Example usage:
   smbpe.exe -r "LDA $01; STA $02; RST"
Note: The assembly code should be enclosed in quotes. Lines are separated by semicolons, this is only for raw mode and not consistent across other modes.

File mode is used for running complete programs stored in .smbp files. It takes the file path as input, loads the program, and runs it in the emulator. This mode is suitable for running larger programs and games.
Example usage:
   smbpe.exe -f path/to/your/program.smbp
   smbpe.exe -g -f path/to/your/program/with/graphics.smbp 
Note: Ensure the file path is correct and the file exists. If file has graphics, specify the -g flag. If programs that have graphics are not used with the -g flag, the graphics will not be rendered.
Additional flags:
   -d : Enables debug mode, providing detailed output for debugging purposes.
   -h : Displays help information about the emulator's usage and options.
   -g : Enables graphics mode, allowing the rendering of graphical elements if the program supports it.
------ End Of Emulation ------
